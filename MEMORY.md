# KnowledgeHub AI — Error Memory & Lessons Learned

This file documents every error encountered during development, its root cause, the fix applied, and what should be checked proactively in future projects.

---

## 1. Missing PostCSS config for Tailwind v4

**Error**: Frontend `npm run build` fails — `@import "tailwindcss"` in `globals.css` can't be processed.

**Root cause**: Tailwind CSS v4 requires `@tailwindcss/postcss` to be explicitly registered in a `postcss.config.mjs` file. We had `tailwindcss` and `@tailwindcss/postcss` in `package.json` but no PostCSS config.

**Fix**: Created `frontend/postcss.config.mjs`:
```js
export default {
  plugins: {
    '@tailwindcss/postcss': {},
  },
};
```

**Lesson**: When using Tailwind CSS v4, always create `postcss.config.mjs` before the first build. This is not auto-generated by Next.js scaffolding when you manually set up the project.

## 2. Missing frontend/public/ directory

**Error**: Docker build fails because the frontend Dockerfile does `COPY --from=builder /app/public ./public` but the directory doesn't exist.

**Root cause**: The `public/` directory is standard for Next.js static assets but was not created.

**Fix**: Created `frontend/public/.gitkeep`.

**Lesson**: When scaffolding a Next.js project, always create the `public/` directory. Dockerfile references to it will fail silently if it's missing.

## 3. Missing Docker infrastructure directories

**Error**: Docker Compose references volumes that don't exist:
- `./docker/nginx/conf.d/`
- `./docker/nginx/ssl/`
- `./docker/postgres/init/`

**Root cause**: These directories were listed in `docker-compose.yml` volumes but never created in the repo.

**Fix**: Created all three directories. Docker creates them as root-owned empty dirs if they don't exist, but explicit creation avoids confusion.

**Lesson**: Always create directories referenced in compose volume mounts. Even though Docker auto-creates them, they should be in the repo.

## 4. Frontend Dockerfile missing build-time env vars

**Error**: `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_WS_URL` set in docker-compose at runtime, but Next.js inlines `NEXT_PUBLIC_*` at **build time**.

**Root cause**: Runtime `environment` in docker-compose doesn't affect built JS bundles. Next.js replaces `process.env.NEXT_PUBLIC_*` with the literal value during `npm run build`.

**Fix**: Added `ARG` declarations at the top of the Dockerfile and `ENV` assignments in the builder stage. Docker Compose passes these as `build.args`.

**Lesson**: `NEXT_PUBLIC_*` env vars must be available at build time, not runtime. Use Docker build args.

## 5. Frontend volume mount shadows standalone build output

**Error**: Frontend container starts but serves nothing — 404 or blank page.

**Root cause**: The volume mount `- ./frontend:/app` replaces the entire `/app` directory (including `.next/standalone`, `.next/static`, `node_modules`) with the host's source code. The multi-stage Docker build output is invisible.

**Fix**: Removed the `volumes` block from the frontend service in docker-compose.

**Lesson**: Don't use volume mounts on containers that rely on multi-stage build output. The volume shadows everything. Use mounts only for development hot-reload setups (and adjust the command accordingly).

## 6. Import at bottom of file (skeleton.tsx)

**Error**: No build error, but violates convention and can confuse bundlers/linters.

**Root cause**: `import { cn } from "@/lib/utils"` was placed at line 4 (after the component) instead of the top.

**Fix**: Moved import to line 1.

**Lesson**: Always place imports at the top of the file. This is standard practice and some tools flag it.

## 7. Invalid semicolons in .gitignore

**Error**: Git doesn't ignore `__pycache__/`, `node_modules/`, `.next/`, `.env`, etc.

**Root cause**: The `.gitignore` used semicolons to separate patterns on one line:
```
__pycache__/; *.py[cod]; *.egg-info/; .venv/; venv/
```
Git treats each line as a single pattern. The entire line was interpreted as one pattern: `__pycache__/; *.py[cod]; *.egg-info/; .venv/; venv/`. Nothing was actually ignored.

**Fix**: Rewrote each pattern on its own line.

**Lesson**: `.gitignore` does NOT support semicolons as separators. One pattern per line. Always verify `.gitignore` after generation.

## 8. Card component missing onClick prop

**Error**: TypeScript build error:
```
Property 'onClick' does not exist on type 'IntrinsicAttributes & { className?: string; children: ReactNode; }'.
```

**Root cause**: The `Card` component's props type only included `className` and `children`, but `admin/page.tsx` passed `onClick` to use cards as clickable navigation.

**Fix**: Added `onClick?: React.MouseEventHandler<HTMLDivElement>` to Card props and passed it through to the underlying `<div>`.

**Lesson**: When building UI components, consider common HTML event props (`onClick`, `onMouseEnter`, etc.) or extend native HTML element props via `ComponentPropsWithoutRef` or similar.

## 9. Qdrant healthcheck uses wget (not available in image)

**Error**: Docker marks qdrant as unhealthy.
```
✘ Container knowledgehub-qdrant Error dependency qdrant failed to start
```

**Root cause**: The healthcheck used `wget --spider http://localhost:6333/health`, but the `qdrant/qdrant` Docker image does **not** include `wget` or `curl`. The command fails with "command not found".

**Fix attempt 1**: Changed to bash `/dev/tcp` healthcheck:
```yaml
test: ["CMD", "bash", "-c", "exec 3<>/dev/tcp/127.0.0.1/6333"]
```
But this was initially set as `CMD-SHELL` which runs under `sh` (dash on Debian), and dash doesn't support `/dev/tcp`.

**Fix attempt 2**: Changed to `CMD` form with explicit `bash -c`, but determined the whole healthcheck approach was fragile.

**Final fix**: Disabled healthcheck entirely (`healthcheck: disable: true`), changed backend/arq-worker `depends_on` for qdrant to `condition: service_started`. Backend connects lazily so timing is fine.

**Lesson**: Never assume CLI tools like `wget` or `curl` are available in Docker images. Check the base image first. For Qdrant, the image is `debian:12-slim` which has minimal tooling. When a healthcheck can't be reliably implemented, disable it and use `condition: service_started`.

## 10. CMD-SHELL runs under sh, not bash

**Error**: Bash-specific `/dev/tcp` syntax didn't work in healthcheck.

**Root cause**: `CMD-SHELL` in Docker healthcheck runs the command with `sh -c`. On Debian, `/bin/sh` is `dash`, which does not support bash-specific features like `/dev/tcp`.

**Fix**: Used `CMD` form instead: `["CMD", "bash", "-c", "exec 3<>/dev/tcp/127.0.0.1/6333"]`. But this was ultimately replaced with healthcheck disabled.

**Lesson**: `CMD-SHELL` uses `sh`, not `bash`. If you need bash features, use the `CMD` exec form: `["CMD", "bash", "-c", "..."]`.

## 11. Frontend port 3000 conflict

**Error**: Docker networking error:
```
Bind for 0.0.0.0:3000 failed: port is already allocated
```

**Root cause**: A previous container or process was already using host port 3000.

**Fix**: Changed frontend host port mapping from `3000:3000` to `2002:3000`.

**Lesson**: Always check for port conflicts before starting containers. Use `docker ps` or `lsof -i :PORT` to debug.

## 12. NEXT_PUBLIC_API_URL set to localhost

**Error**: Registration API calls fail with network errors from the browser.

**Root cause**: `NEXT_PUBLIC_API_URL` was baked as `http://localhost:8000`. When a user accesses the frontend from their browser at `http://192.168.3.118:2002`, `fetch("http://localhost:8000/...")` tries to connect to port 8000 on the **user's local machine**, not the server.

**Fix**: Changed to `http://192.168.3.118:8000` (the server's actual IP). Updated docker-compose build args, default in Dockerfile, `.env`, `.env.example`, and the fallback in `lib/api.ts`.

**Lesson**: In containerized deployments where the frontend is served to remote browsers, `NEXT_PUBLIC_API_URL` must be the server's **externally reachable** address, not `localhost`. `localhost` only works when the browser is on the same machine as the backend.

## 13. AuthGuard blocks /register page

**Error**: Navigating to `/register` immediately redirects to `/login`.

**Root cause**: The `AuthGuard` component only allowed `/login` as a public path. Any unauthenticated user hitting any other path was redirected to `/login`.

**Fix**: Added `/register` to the public paths whitelist in `AuthGuard`.

**Lesson**: When adding new authentication-free pages, update the AuthGuard's whitelist immediately.

## 14. Registration error handling too generic

**Error**: User sees "Registration failed" with no details about what went wrong.

**Root cause**: The catch block in `register/page.tsx` was:
```typescript
catch { toast.error("Registration failed"); }
```

**Fix**: Changed to show the actual error message:
```typescript
catch (err: any) { toast.error(err.message || "Registration failed"); }
```

**Lesson**: Always surface actual error details in development. Generic error messages mask the root cause.

## 15. Missing PYTHONPATH on backend container

**Error**: Alembic migration fails:
```
ModuleNotFoundError: No module named 'app'
```

**Root cause**: The backend container runs with WORKDIR `/app`, but when executing `alembic upgrade head` (via `docker compose exec`), Python's `sys.path` doesn't include `/app` because the module is loaded by Alembic's internal loader, not by Python's normal import system.

**Fix**: Added `PYTHONPATH: /app` to the backend service's `environment` in docker-compose.

**Lesson**: For containers running Python applications with Alembic, always set `PYTHONPATH` explicitly. The WORKDIR alone may not be sufficient when running subcommands like `alembic`.

## 16. Missing qualname in alembic.ini

**Error**: Alembic migration fails:
```
File "/usr/local/lib/python3.12/logging/config.py", line 247, in _install_loggers
    qn = section["qualname"]
KeyError: 'qualname'
```

**Root cause**: The `[logger_alembic]` section in `alembic.ini` was missing the `qualname` key. Python's `logging.config.fileConfig()` requires `qualname` for each logger section.

**Fix**: Added `qualname = alembic` to the `[logger_alembic]` section.

**Lesson**: When using `fileConfig()` with Python logging (as Alembic's default `env.py` does), every logger section must have a `qualname` key. Check the generated `alembic.ini` for missing required fields.

**Root cause**: Alembic migration was never run. The database exists but has no tables.

## 18. No SSH key in CI/workspace environment

**Error**: `git push` fails:
```
git@github.com: Permission denied (publickey).
```

**Root cause**: The workspace environment has no SSH key configured for GitHub.

**Fix**: Used GitHub Personal Access Token (classic, with `repo` scope) for pushes. The PAT was used via HTTPS remote URL and reverted back to SSH after each push.

**Lesson**: In environments without SSH keys, use a classic PAT with `repo` scope. Fine-grained PATs need explicit repo authorization and specific permissions (Contents: write). Keep the PAT secure.

## 19. Git permissions on Ubuntu server

**Error**: `sudo git pull` fails with SSH key error.

**Root cause**: `sudo` runs as root, which doesn't have the user's SSH key at `/home/nitesh/.ssh/id_ed25519`.

**Fix**: Use:
```bash
sudo GIT_SSH_COMMAND="ssh -i /home/nitesh/.ssh/id_ed25519" git pull
```

**Lesson**: When using `sudo` with git, explicitly pass the SSH key path. Alternatively, configure root's SSH config or use a PAT/HTTPS remote.

## 20. First user not auto-promoted to admin

**Error**: First registered user is `employee` role, can't access admin panel.

**Root cause**: The `AuthService.register()` method always set `role = UserRole.employee`.

**Fix**: Added a check: if no users exist in the database, assign `UserRole.admin` to the first registered user.

**Lesson**: When building auth systems, consider the first-user experience. Auto-promotion to admin avoids manual database intervention.

## Assumptions That Caused Errors

| Assumption | Reality | Impact |
|---|---|---|
| Tailwind v4 works without PostCSS config | Requires explicit `postcss.config.mjs` | Build failure |
| Docker images include `wget`/`curl` | Minimal images often don't | Healthcheck failure |
| `CMD-SHELL` uses `bash` | Uses `sh` (dash) | Bash-specific syntax breaks |
| `NEXT_PUBLIC_*` vars can be runtime env | Inlined at build time | Wrong API URL in browser |
| Volume mounts preserve build output | Shadow it completely | Frontend serves nothing |
| `.gitignore` supports semicolons | One pattern per line only | Nothing ignored |
| `localhost` works in browser JS | Points to user's machine, not server | Network errors |
| Alembic's `env.py` finds app module | Needs `PYTHONPATH` | Migration fails |
| Alembic's ini template has all fields | Missing `qualname` | Logging config crashes |
| Qdrant healthcheck can use HTTP tools | No wget/curl in image | Unhealthy container |
| SSH key works with `sudo` | Root doesn't have user's keys | Git pull fails |

## 21. passlib incompatible with bcrypt >= 4.1

**Error**: Backend logs show:
```
WARNING:passlib.handlers.bcrypt:(trapped) error reading bcrypt version
AttributeError: module 'bcrypt' has no attribute '__about__'
```
Registration and login silently fail or produce corrupted password hashes.

**Root cause**: `passlib` uses `bcrypt.__about__` to detect the bcrypt version. The `bcrypt` library removed `__about__` in version 4.1. `requirements.txt` had `passlib[bcrypt]>=1.7.4` which pulls in the latest `bcrypt`, but nothing pinned it below 4.1.

**Assumption**: `passlib[bcrypt]` would pull a compatible version of `bcrypt`. In reality, pip installs the latest `bcrypt` regardless of passlib's requirements.

**Fix**: Added `bcrypt>=4.0.0,<4.1.0` to `requirements.txt` to pin below the breaking version.

**Lesson**: When using `passlib[bcrypt]`, always pin `bcrypt` explicitly. The extra `[bcrypt]` does NOT constrain the bcrypt version. Check the passlib changelog for bcrypt compatibility before each upgrade.

## 22. Token not stored before /users/me call in login flow

**Error**: Login returns HTTP 200 with a valid token, but immediately redirects back to login page. Backend logs show `POST /auth/login` → 200, then `GET /users/me` → 401.

**Root cause**: The login page stores the token in localStorage via `setAuth()` **after** calling `/api/v1/users/me`. The `api.ts` request function reads the token from `localStorage.getItem("token")`, which is still `null` during the `/users/me` call. The request goes out without an `Authorization` header, so the backend returns 401.

**Assumption**: `setAuth()` was called before the `/users/me` fetch, so the token would be available. In reality, the token variable existed but wasn't persisted to localStorage yet.

**Fix**: Added `localStorage.setItem("token", access_token)` before the `/api/v1/users/me` call in `frontend/app/login/page.tsx`.

**Lesson**: After any API call that returns a token, persist it to localStorage immediately before making any subsequent authenticated requests. The `api.ts` client reads from localStorage on every call — it does not accept a token parameter.

## 23. Admin health page linked to "#" — no route existed

**Error**: Clicking "System Health" card in admin dashboard did nothing (stayed on same page).

**Root cause**: The admin page at `frontend/app/admin/page.tsx:16` had `href: "#"` for both "System Settings" and "System Health" cards. The `onClick={() => router.push(href)}` with `href="#"` navigates to the current page URL with `#` appended, effectively a no-op. No `/admin/health` route existed.

**Assumption**: Admin pages for health and settings would be built later, but `"#"` was used as a placeholder without even creating the route file. The user expected working navigation upon clicking the card.

**Fix**: Created:
- Backend: Enhanced `GET /api/v1/health/` to check Postgres, Redis, Qdrant, and Ollama connectivity
- Frontend: Created `/admin/health` page with service status cards using `@tanstack/react-query` with 30s auto-refresh
- Updated admin card href from `"#"` to `"/admin/health"`

**Lesson**: Never use `"#"` as a placeholder href for navigation cards. Either create the target page, disable the card, or route to a proper "coming soon" page. Every clickable element in the UI should lead to a valid route.

## Pre-Flight Checklist for Future Projects
